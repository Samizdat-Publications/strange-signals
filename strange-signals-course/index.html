The Scene

Before we start

This is a six-part tour of a real codebase called Strange Signals — an interactive map of paranormal sightings. The goal isn’t to make you a programmer. It’s to give a “vibe coder” — someone who builds with AI but doesn’t read code — the vocabulary and mental map to steer a project like this with confidence.

The big idea: it’s a pipeline

Here is the single most important thing to understand about Strange Signals: it is not one program. It’s a pipeline of small programs that each do one job. The map you see is the last step. Everything before it happened on someone else’s computer, was saved as a file, and shipped to your browser.

Two terms worth pausing on. A JSON file is just a text file full of records — the whole dataset is one big one. And a Web Worker is a helper inside your browser that can chew through that file without freezing the page. Module 3 goes deep on the worker; for now, just know it exists.

!
Aha! The app has no backend.

There is no backend server computing anything in real time. The JSON file was pre-computed on someone’s laptop, committed to disk, and is served as a static file. Every feature — filters, correlations, even the AI assistant — runs on your computer. No server costs, no API downtime, no database. But also: no live data. Every time the datasets update, someone has to rebuild and re-publish the JSON.

That’s the tradeoff worth internalizing. Pre-computed data is fast, cheap, and offline-capable — but rigid. Live data is flexible but requires a server, a database, and someone to keep it all running. Strange Signals chose the first path. Knowing why helps you make the same call on your own projects.

How they pass notes: the bubble trick

Here's the architectural idea that matters most. In a writers' room, not every scribbled note on a whiteboard is for public consumption — most of it is private to one writer. A few deliberate things get taped to the door so anyone can read them.

Strange Signals does the same thing with an IIFE. The entire Lead script is wrapped in a bubble — a private room. Variables inside stay inside. At the very end of the file, the Lead picks a small set of functions and tapes them to window.StrangeSignals so the other files can reach them.

The punchline

The first five stops all happen on a developer's laptop before the app is ever shipped. Only the last two — downloading the JSON and parsing it — happen on your machine.

Trick #1: the packing trick

Imagine labeling every single box in the warehouse with the words "this is a box" on all six sides. Useful? No. Wasteful? Absolutely. That's what a verbose JSON record does — it repeats the column names on every single row.

With 258,000 records, repeating "latitude" and "longitude" and "category" over and over adds up to many megabytes of dead weight. So this app strips the labels and stores every record as a plain list, knowing the code on the other end can figure out the order.

A

Verbose (what most AI writes first)

{ "latitude": 39.12,
  "longitude": -84.56,
  "category": "UFO/UAP",
  "date": "2020-01-15",
  "location": "Cincinnati, OH" }

Every column name written out every time. Clear to read, huge on disk.

B

Compact (what Strange Signals does)

[39.12, -84.56, 0,
 "2020-01-15",
 "Cincinnati, OH",
 "triangle",
 "Bright light..."]

A plain list. Position 0 is always latitude. Position 1 is always longitude. Roughly half the file size.

!
Aha! Array positions are the column names.

The code knows position 0 = lat, position 1 = lon, position 2 = category. The F = {LAT:0, LON:1, ...} cheat sheet from Module 2 is exactly what reads these lists. The column names live in one place — in the code — not 258,000 times on disk.

How the packing actually happens

Here's the Python that does the squeezing. It runs once, on the developer's laptop, when the data pipeline rebuilds.

for r in df.itertuples(index=False):
    records.append([
        float(r.latitude),
        float(r.longitude),
        int(r.cat),
        r.date_str if r.date_str != "nan" else "",
        r.loc,
        r.sub if r.sub != "nan" else "",
        r.desc if r.desc != "nan" else "",

output = {
    "categories": ["UFO/UAP", "Bigfoot/Sasquatch", "Haunted Place"],

PLAIN ENGLISH

Start with an empty list. Every sighting in the warehouse gets added to it.

Walk every row of the cleaned DataFrame one at a time. Using vectorized iteration makes this pass fast.

For each row, throw away the column names. Keep the values in a fixed order: latitude, longitude, category code, date, location, subcategory, description.

Empty strings replace missing values so downstream code never has to check for "nan".

Wrap the giant list in a tiny header that tells the browser what each position means. This header is the decoder ring — fields[0] is "lat", so data[i][0] is the latitude of record i.

Trick #2: the unfreeze trick

Back to the warehouse. Your main thread is the shopkeeper at the counter. Smiling at customers. Ringing things up. If the shopkeeper personally unloads the delivery truck, every customer waits while they sweat through 258,000 boxes.

So you hire a dock worker — a Web Worker. They open the truck around back, sort the pallets into three bins (UFO, Bigfoot, Haunted), and pass finished boxes up to the counter in small batches. The shopkeeper keeps smiling. Nobody notices the unloading.

1
Validate and categorize in one pass

The worker throws out any record with a bad latitude, longitude, or category code — and groups the survivors into three lists (one per category).

2
Ship results in 5,000-record batches

Each small chunk crosses the thread boundary via postMessage. Small chunks copy fast. A giant blob would cause a visible hiccup on the main thread.

The entire worker, in 30 lines

CODE 
// Web Worker: decode, parse, validate, categorize — all off main thread.
// Sends results back in small batches so the main thread never blocks on structured clone.
self.onmessage=function(e){
  try{
    const text=new TextDecoder().decode(e.data);
    const json=JSON.parse(text);
    const raw=json.data;
    if(!Array.isArray(raw)||!raw.length){
      self.postMessage({type:'error',error:'Invalid or empty data'});
      return;
    }
    // Filter + categorize in one pass
    const cats=[[],[],[]];
    for(let i=0;i<raw.length;i++){
      const r=raw[i];
      if(!Array.isArray(r)||r.length<7)continue;
      if(typeof r[0]!=='number'||isNaN(r[0]))continue;
      if(typeof r[1]!=='number'||isNaN(r[1]))continue;
      if(r[2]<0||r[2]>2)continue;
      cats[r[2]].push(r);
    }
    const total=cats[0].length+cats[1].length+cats[2].length;
    // Send back in small batches to avoid structured clone blocking main thread
    const BATCH=5000;
    for(let cat=0;cat<3;cat++){
      for(let i=0;i<cats[cat].length;i+=BATCH){
        self.postMessage({type:'batch',cat:cat,records:cats[cat].slice(i,i+BATCH)});
      }
    }
    self.postMessage({type:'done',total:total});
  }catch(err){
    self.postMessage({type:'error',error:err.message});
  }
};
PLAIN ENGLISH

This worker sits quietly and waits for the main thread to hand it a file. When that happens, this function runs.

Turn the raw bytes into text, then turn the text into a JavaScript object.

Sanity check: is there actually data here? If not, send an error back and bail out.

Three empty buckets — one per category (UFO, Bigfoot, Haunted).

Walk every record. Skip the broken ones: wrong shape, bad latitude, bad longitude, or a category code outside 0 through 2.

Drop each valid record into the bucket named by its category code. All three buckets fill up in a single pass.

Now ship results back — 5,000 records at a time.

Each chunk is small enough that the browser's structured clone copy is invisible to the user.

When the last batch is out the door, send a "done" message with the final tally.

If anything blew up, send the error back to the main thread instead of crashing silently.

What the main thread actually does

Now that the worker is doing the heavy lifting, the shopkeeper's job is almost embarrassingly small: receive a box, drop it in the right bin, update the progress bar, smile at the next customer. Repeat.

1
A batch message arrives

The worker calls postMessage({type:'batch', cat:0, records:[...5000 records...]}). The main thread's listener wakes up for a fraction of a second.

2
Append to the right bucket

The main thread has three arrays in memory: catArrays[0] (UFO), catArrays[1] (Bigfoot), catArrays[2] (Haunted). It pushes the batch onto the one matching cat. That's it.

3
Nudge the progress bar, then back to idle

The UI updates. The main thread returns to its normal duties — handling clicks, scrolls, and redraws — until the next batch lands.

4
"done" arrives, the map draws

When the worker signals completion, the main thread kicks off the rendering pipeline — which is Module 4's whole job.

Two tricks every big-data browser app should use

{ }

Compact array format

Store records as plain lists, not objects with repeated keys. Half the file size. The field-order header is your decoder ring.

W

Web Worker + batched postMessage

Parse and validate off the main thread. Ship results back in small chunks so the copy across threads is never a visible hiccup.

!
Why this matters when you're using AI

Ask AI to "load a big JSON file in the browser" and by default it'll hand you fetch(url).then(r => r.json()) on the main thread — which freezes the UI for big files. Now you know the names to ask for: compact array format and parse it in a Web Worker with batched postMessage. That vocabulary is the difference between a snappy app and a frozen tab.

04

Rendering 258K Pins Fast

A stage magician's repertoire — six tricks that keep the page feeling instant.

The problem of the frozen page

Your browser has one painter. One. Every pixel on the page — the pins, the popup, the sidebar charts, the animated loading text — is drawn by the same single worker. Ask it to place 258,000 pins at once and it stops answering the door for ten seconds. The page freezes, the cursor stalls, clicks are ignored.

Modern web apps don't get around this by being clever with the paint. They get around it by never asking for more than the painter can handle at once. The main thread is a single lane — the trick is to never block it for long.

A stage magician doesn't actually saw the assistant in half. They make you look at the left hand while the right hand does the work. Strange Signals uses six such tricks — none of them drawing faster, all of them hiding the work.

*
The timeline picture

One unbroken 10-second render = a frozen gray zone. 52 batches of 5,000 with a breath in between = the page stays alive the whole time. Same total work. Totally different feel.

Trick #1 — Batched rendering

The first illusion is the showpiece. renderMarkers loops over 258K records, but it only creates 5,000 markers at a time. Then it calls setTimeout(addBatch, 0) — a polite way of saying "do this next, but not right now."

That tiny gap is the whole trick. In it, the browser gets a turn to paint a frame, handle a click, let the event loop breathe. Then our next slice runs. Repeat 52 times. The user sees pins appear in pulses instead of waiting in the dark.

CODE 
let markerRenderGen=0;
function renderMarkers(){
  const myGen=++markerRenderGen;
  const BATCH=5000;
  for(let i=0;i<3;i++){
    if(!filteredCat[i].length)continue;
    const group=createCluster(i);
    const data=filteredCat[i];
    let idx=0;
    function addBatch(){
      if(markerRenderGen!==myGen)return; // stale render — abort
      const end=Math.min(idx+BATCH,data.length);
      const batch=[];
      for(;idx<end;idx++)batch.push(createMarker(data[idx]));
      group.addLayers(batch);
      if(idx<data.length)setTimeout(addBatch,0);
    }
    addBatch();
    group.addTo(map);
    clusterGroups[i]=group;
  }
}
PLAIN ENGLISH

A counter that ticks up every time a render starts. It lives outside the function so it survives between renders.

The entry point. This is what runs when the user clicks "Markers."

Capture my number before I start work. If a newer render comes in, it'll bump the counter past mine — and I'll know to bow out.

Do 5,000 markers per slice. Picked empirically: big enough that the loop overhead is negligible, small enough that one slice finishes in a few milliseconds.

Loop over the three categories (UFO, Bigfoot, Haunted). Skip any that have no filtered data.

Set up a Leaflet cluster group to hold this category's pins.

The inner worker function. This is the one that gets called over and over.

Am I stale? If someone newer has started, abandon the rest of my work immediately. No cleanup needed.

Grab the next 5,000 markers (or whatever's left).

Build them into a temporary array, then add them all at once — one DOM touch instead of 5,000.

Still more to do? Queue up the next batch with setTimeout. The 0 means "right after the browser catches up on everything else."

Kick it off the first time, attach the cluster to the map, and stash a reference so we can remove it later.

Screen 3 — The other five tricks

Batching is just one illusion in the magician's bag. The rest of the app leans on five more, each suited to a different kind of heavy lifting. Learn their names — the next time an AI-built app feels janky, you'll know which one to ask for.

1

Batched rendering

What: Chop a big loop into 5,000-item slices with a setTimeout between each.

When to ask for it: any loop over 10K+ items that runs on the main thread.

2

Spatial index caching

What: Build an expensive spatial index once, reuse it on every hover and recount.

When to ask for it: the same input keeps being asked the same question (hover-hover-hover).

3

Canvas renderer

What: Tell Leaflet preferCanvas: true. Pins become pixels on one canvas, not 258K DOM elements.

When to ask for it: drawing thousands of small graphical things.

4

MarkerCluster

What: A Leaflet plugin that lumps nearby pins into one circle-with-a-count and only renders what's visible at this zoom.

When to ask for it: data density is way higher than the screen's pixel density.

5

Debounced zoom

What: Ignore every zoomend event except the last — wait for 300ms of quiet before re-rendering.

When to ask for it: an event fires in rapid bursts and each run is expensive.

6

Web Worker

What: Hand the hex-count math to hex-worker.js — a parallel thread — so the main thread stays free to paint.

When to ask for it: pure math work that takes more than ~50ms.

Trick #7 — The stale render problem

Here's the subtle bit — the one AI almost always forgets. Batched rendering takes time. While one render is mid-way through its 52 slices, what happens if the user clicks "apply filter" again, with different inputs? Now there are two batch sequences running, both writing to the same cluster groups. Pins from the old filter mingle with pins from the new one. Chaos.

The fix is one integer: markerRenderGen. Every new render bumps the counter and captures the old value as myGen. At the top of every batch, it checks: "Am I still the newest render?" If a newer one started, the stale batch quietly bows out. No cleanup. No coordination. Just one number.

1
First click lands. markerRenderGen goes 0 → 1.

Render A grabs myGen = 1, starts chewing through 52 batches.

2
Halfway through, the user clicks again. markerRenderGen goes 1 → 2.

Render B grabs myGen = 2, starts its own 52 batches.

3
Render A's next batch wakes up.

It checks: markerRenderGen (2) !== myGen (1). It returns. Silently. Its remaining batches never run.

4
Render B finishes clean.

Only the newest filter's pins make it to the map. No mingling. No bug report.

!
Aha! Long async work needs a stale-check.

Any time you kick off work that spans multiple ticks of the event loop, you need a way to detect it's been superseded. A generation counter is the cheapest version of that idea — one integer, one check per slice, done. Whenever you ask AI for batched, debounced, or streamed work, also ask: "What happens if a new one starts before the old one finishes?"

The Correlation Math

How the app separates a dramatic headline from actual evidence.

The Question the App Is Really Answering

You zoom into the correlation view and see it: a Pearson r of 0.71 between UFO hotspots and military bases. The number feels huge. It must mean something.

Before you post the screenshot to Reddit, Strange Signals shuffles the data 999 times, recomputes the correlation on each scramble, and asks: "How often does a r ≥ 0.71 happen by pure dumb luck?" If the answer is "less than 1 in 1000," you have something. If it's "1 in 5," you have a coincidence.

This module is the math that separates the two.

⚖
The courtroom frame

The prosecution shows the jury a dramatic chart. The defense stands up and asks a single question: "Could this pattern happen by pure random chance?" If "yeah, pretty often," the case collapses. That question is what a p-value answers, and every correlation Strange Signals shows you has one attached.

The comparison is the point

You have three categories of sightings and a dozen overlay datasets. The question is never "are there UFO hotspots?" — you can see that on the markers map. The question is always comparative:

◎
The question every correlation answers

"Is there a suspicious overlap between UFO hotspots and some other thing — military bases, earthquakes, caves, haunted places?" Correlation is the math of overlap.

Step 2: Pearson r, the Correlation Number

Pearson r asks one intuitive question: "When X is above its average, is Y also above its average?" If yes across the board, r approaches +1. If X goes up while Y goes down, r approaches −1. If there's no pattern, r hovers near 0.

CODE — strange-signals.js 
function pearsonR(xArr,yArr){
  const n=xArr.length;
  if(n<5)return NaN;
  const mx=d3.mean(xArr),my=d3.mean(yArr);
  let num=0,dx2=0,dy2=0;
  for(let i=0;i<n;i++){
    const dx=xArr[i]-mx,dy=yArr[i]-my;
    num+=dx*dy;dx2+=dx*dx;dy2+=dy*dy;
  }
  const den=Math.sqrt(dx2*dy2);
  return den>0?num/den:NaN;
}
PLAIN ENGLISH

Take two parallel lists of numbers.

Refuse to answer if we have fewer than 5 data points. Any correlation on 4 points is noise.

Compute the average of each list. Correlation is about deviations from the average, not raw values.

Set up three running totals: the "agreement" number and two "spread" numbers.

For each hex: how far above or below its average is this X? Same for Y.

Multiply the two deviations. If both are positive (both above average) or both negative (both below), this adds a positive number. If they disagree in sign, it subtracts. This is the heart of the formula.

Normalize by how spread out each list is, so the answer lives between −1 and +1 no matter the scale.

If one list has zero variation (all the same value), correlation is undefined — return NaN.

A quick sanity scale

0
r ≈ 0.0 to 0.2

Essentially unrelated. Whatever you're seeing is noise.

~
r ≈ 0.2 to 0.5

Weak-to-moderate link. Interesting, but don't bet the farm.

+
r ≈ 0.5 to 0.8

Meaningful co-movement. The two lists really do rise and fall together.

!
r ≈ 0.8+

Suspiciously tight. Go look for a cause — or a confounder.

Step 3: The Permutation P-Value

The defense attorney stands up. "r = 0.71 is a nice number, but so what? Could random noise have produced it too?" The app answers by simulation.

Imagine writing every Y value on an index card. Shuffle the cards. Now X and Y have no real relationship — Y is in random order. Compute r on this scrambled pair. Save it. Shuffle again. Recompute. 999 times. You've built a histogram of "correlations that happen by pure chance." Compare your observed r to that histogram.

CODE — strange-signals.js 
function permutationPValue(xArr,yArr,observedR,nPerms){
  nPerms=nPerms||999;
  if(xArr.length<5||isNaN(observedR))return 1;
  let extremeCount=0;
  const yCopy=yArr.slice();
  for(let p=0;p<nPerms;p++){
    for(let i=yCopy.length-1;i>0;i--){
      const j=Math.floor(Math.random()*(i+1));
      [yCopy[i],yCopy[j]]=[yCopy[j],yCopy[i]];
    }
    const permR=pearsonR(xArr,yCopy);
    if(!isNaN(permR)&&Math.abs(permR)>=Math.abs(observedR))extremeCount++;
  }
  return(extremeCount+1)/(nPerms+1);
}
PLAIN ENGLISH

Do 999 scrambles by default. That many shuffles gives enough resolution to tell tiny p-values apart.

Bail out cleanly if there isn't enough data or the observed r failed.

Keep a counter: how many shuffles produced a correlation at least as extreme as the real one?

Don't destroy the real Y array. Make a working copy we can shuffle.

Outer loop: do the whole thing 999 times.

Inner three lines: the Fisher-Yates shuffle. Walk from the end to the start and swap each item with a random earlier one. This produces a uniform random permutation in one pass.

Correlate X with scrambled Y. Any pattern here is pure coincidence — the pairing is random.

Count the scrambles that produced a correlation as strong or stronger than what we actually observed. Math.abs because we care about "strong" in either direction.

Fraction of scrambles that beat or matched the real data. That fraction is the p-value. The +1s are a standard correction so p is never exactly zero.

Significance stars, decoded

*** p < 0.001 — would happen by chance less than 1 in 1000 times. Strong evidence.
** p < 0.01 — less than 1 in 100. Solid evidence.
* p < 0.05 — less than 1 in 20. The conventional bar for "publishable."
n.s. p ≥ 0.05 — "not significant." Could be coincidence; proceed with caution.

These are significance stars — shorthand you'll see in any academic paper. Strange Signals prints them right next to the r value.

✨
Aha! Scrambling is the null hypothesis made concrete.

The null hypothesis is the boring default: "there's no real link — any pattern is luck." A permutation test doesn't derive a p-value from a textbook formula — it simulates what chance looks like by actually breaking the link 999 times. If you can shuffle an array, you never need to open a statistics textbook again.

SIGNAL AI — The Agent Loop

How a text-only model gets hands — 80 lines of JavaScript that turn Claude into an air-traffic controller for your map.

What just happened?

You type "show me UFO hotspots near military bases in the 90s" into SIGNAL's chat box. Three seconds later the map has zoomed, the year filter is 1990–1999, the military overlay is on, and the heatmap is rendering. You didn't click anything.

Picture an LLM as an air-traffic controller in a tower with binoculars and a radio. The controller cannot fly the planes — all they can do is see (look through the binoculars) and talk (issue instructions over the radio). Strange Signals' AI is the same: Claude cannot touch the map directly. It can only ask to see something or do something, and the browser runs each request and reports back.

The fetch call, demystified

There is no backend. SIGNAL's entire "server" is this one function — the browser talks to Anthropic directly, using the user's own API key stored in localStorage. Here is the setup half of the loop:

CODE 
async function runConversationLoop(){
  var apiKey=getApiKey();
  var model=getModel();

  // Mark system prompt + last tool as cacheable (ephemeral, ~5 min TTL)

      model:model, max_tokens:4096,
      system:cachedSystem, tools:cachedTools,

PLAIN ENGLISH

Grab the API key the user pasted in (stored in localStorage) and the model they picked from a dropdown.

Mark the system prompt as cacheable. This is prompt caching — Anthropic will hold this text in cache for about five minutes so you are not charged for it every turn.

Do the same for the last entry in the tool catalog. Marking one tool automatically caches everything up to and including it.

Loop up to ten times. Hard cap — enough for multi-step plans but bounded so a confused model cannot run away with your bill.

Build the request body. The whole conversation goes in the messages array every single time — Claude has no memory between calls, so the array is the memory.

Streaming on, so tokens show up in the chat as they arrive.

Call fetch straight to Anthropic. No backend. The browser is the server.

The API key goes in the x-api-key header. Whoever opens DevTools on this page can see it — fine for a personal tool, never for a shipped product.

The anthropic-dangerous-direct-browser-access header is the opt-in flag that says "yes, we really are calling the API straight from the browser." Without it, CORS would block the request.

Serialize the body to JSON and send it.

In real life there is also a retry block handling 429 / 529 errors — if the API says "too fast," the app sleeps and tries again.

Parse the streamed response into a clean object with a list of content blocks. Text blocks or tool_use blocks. That is what goes to the next snippet.
