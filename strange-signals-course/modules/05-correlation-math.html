The Correlation Math

How the app separates a dramatic headline from actual evidence.

The Question the App Is Really Answering

You zoom into the correlation view and see it: a Pearson r of 0.71 between UFO hotspots and military bases. The number feels huge. It must mean something.

Before you post the screenshot to Reddit, Strange Signals shuffles the data 999 times, recomputes the correlation on each scramble, and asks: "How often does a r ≥ 0.71 happen by pure dumb luck?" If the answer is "less than 1 in 1000," you have something. If it's "1 in 5," you have a coincidence.

This module is the math that separates the two.

⚖
The courtroom frame

The prosecution shows the jury a dramatic chart. The defense stands up and asks a single question: "Could this pattern happen by pure random chance?" If "yeah, pretty often," the case collapses. That question is what a p-value answers, and every correlation Strange Signals shows you has one attached.

The comparison is the point

You have three categories of sightings and a dozen overlay datasets. The question is never "are there UFO hotspots?" — you can see that on the markers map. The question is always comparative:

The question every correlation answers

"Is there a suspicious overlap between UFO hotspots and some other thing — military bases, earthquakes, caves, haunted places?" Correlation is the math of overlap.

Step 1: Turn a Map Into Two Lists of Numbers

Step 2: Pearson r, the Correlation Number

Pearson r asks one intuitive question: "When X is above its average, is Y also above its average?" If yes across the board, r approaches +1. If X goes up while Y goes down, r approaches −1. If there's no pattern, r hovers near 0.

CODE — strange-signals.js 
function pearsonR(xArr,yArr){
  const n=xArr.length;
  if(n<5)return NaN;
  const mx=d3.mean(xArr),my=d3.mean(yArr);
  let num=0,dx2=0,dy2=0;
  for(let i=0;i<n;i++){
    const dx=xArr[i]-mx,dy=yArr[i]-my;
    num+=dx*dy;dx2+=dx*dx;dy2+=dy*dy;
  }
  const den=Math.sqrt(dx2*dy2);
  return den>0?num/den:NaN;
}
PLAIN ENGLISH

Take two parallel lists of numbers.

Refuse to answer if we have fewer than 5 data points. Any correlation on 4 points is noise.

Compute the average of each list. Correlation is about deviations from the average, not raw values.

Set up three running totals: the "agreement" number and two "spread" numbers.

For each hex: how far above or below its average is this X? Same for Y.

Multiply the two deviations. If both are positive (both above average) or both negative (both below), this adds a positive number. If they disagree in sign, it subtracts. This is the heart of the formula.

Normalize by how spread out each list is, so the answer lives between −1 and +1 no matter the scale.

If one list has zero variation (all the same value), correlation is undefined — return NaN.

A quick sanity scale

0
r ≈ 0.0 to 0.2

Essentially unrelated. Whatever you're seeing is noise.

~
r ≈ 0.2 to 0.5

Weak-to-moderate link. Interesting, but don't bet the farm.

+
r ≈ 0.5 to 0.8

Meaningful co-movement. The two lists really do rise and fall together.

!
r ≈ 0.8+

Suspiciously tight. Go look for a cause — or a confounder.

Step 3: The Permutation P-Value

The defense attorney stands up. "r = 0.71 is a nice number, but so what? Could random noise have produced it too?" The app answers by simulation.

Imagine writing every Y value on an index card. Shuffle the cards. Now X and Y have no real relationship — Y is in random order. Compute r on this scrambled pair. Save it. Shuffle again. Recompute. 999 times. You've built a histogram of "correlations that happen by pure chance." Compare your observed r to that histogram.

CODE — strange-signals.js 
function permutationPValue(xArr,yArr,observedR,nPerms){
  nPerms=nPerms||999;
  if(xArr.length<5||isNaN(observedR))return 1;
  let extremeCount=0;
  const yCopy=yArr.slice();
  for(let p=0;p<nPerms;p++){
    for(let i=yCopy.length-1;i>0;i--){
      const j=Math.floor(Math.random()*(i+1));
      [yCopy[i],yCopy[j]]=[yCopy[j],yCopy[i]];
    }
    const permR=pearsonR(xArr,yCopy);
    if(!isNaN(permR)&&Math.abs(permR)>=Math.abs(observedR))extremeCount++;
  }
  return(extremeCount+1)/(nPerms+1);
}
PLAIN ENGLISH

Do 999 scrambles by default. That many shuffles gives enough resolution to tell tiny p-values apart.

Bail out cleanly if there isn't enough data or the observed r failed.

Keep a counter: how many shuffles produced a correlation at least as extreme as the real one?

Don't destroy the real Y array. Make a working copy we can shuffle.

Outer loop: do the whole thing 999 times.

Inner three lines: the Fisher-Yates shuffle. Walk from the end to the start and swap each item with a random earlier one. This produces a uniform random permutation in one pass.

Correlate X with scrambled Y. Any pattern here is pure coincidence — the pairing is random.

Count the scrambles that produced a correlation as strong or stronger than what we actually observed. Math.abs because we care about "strong" in either direction.

Fraction of scrambles that beat or matched the real data. That fraction is the p-value. The +1s are a standard correction so p is never exactly zero.

Significance stars, decoded

*** p < 0.001 — would happen by chance less than 1 in 1000 times. Strong evidence.
** p < 0.01 — less than 1 in 100. Solid evidence.
* p < 0.05 — less than 1 in 20. The conventional bar for "publishable."
n.s. p ≥ 0.05 — "not significant." Could be coincidence; proceed with caution.

These are significance stars — shorthand you'll see in any academic paper. Strange Signals prints them right next to the r value.

✨
Aha! Scrambling is the null hypothesis made concrete.

The null hypothesis is the boring default: "there's no real link — any pattern is luck." A permutation test doesn't derive a p-value from a textbook formula — it simulates what chance looks like by actually breaking the link 999 times. If you can shuffle an array, you never need to open a statistics textbook again.
