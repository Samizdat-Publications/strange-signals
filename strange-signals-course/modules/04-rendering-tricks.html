Rendering 258K Pins Fast

A stage magician's repertoire — six tricks that keep the page feeling instant.

The problem of the frozen page

Your browser has one painter. One. Every pixel on the page — the pins, the popup, the sidebar charts, the animated loading text — is drawn by the same single worker. Ask it to place 258,000 pins at once and it stops answering the door for ten seconds. The page freezes, the cursor stalls, clicks are ignored.

Modern web apps don't get around this by being clever with the paint. They get around it by never asking for more than the painter can handle at once. The main thread is a single lane — the trick is to never block it for long.

A stage magician doesn't actually saw the assistant in half. They make you look at the left hand while the right hand does the work. Strange Signals uses six such tricks — none of them drawing faster, all of them hiding the work.

Trick #1 — Batched rendering

The first illusion is the showpiece. renderMarkers loops over 258K records, but it only creates 5,000 markers at a time. Then it calls setTimeout(addBatch, 0) — a polite way of saying "do this next, but not right now."

That tiny gap is the whole trick. In it, the browser gets a turn to paint a frame, handle a click, let the event loop breathe. Then our next slice runs. Repeat 52 times. The user sees pins appear in pulses instead of waiting in the dark.

CODE 
let markerRenderGen=0;
function renderMarkers(){
  const myGen=++markerRenderGen;
  const BATCH=5000;
  for(let i=0;i<3;i++){
    if(!filteredCat[i].length)continue;
    const group=createCluster(i);
    const data=filteredCat[i];
    let idx=0;
    function addBatch(){
      if(markerRenderGen!==myGen)return; // stale render — abort
      const end=Math.min(idx+BATCH,data.length);
      const batch=[];
      for(;idx<end;idx++)batch.push(createMarker(data[idx]));
      group.addLayers(batch);
      if(idx<data.length)setTimeout(addBatch,0);
    }
    addBatch();
    group.addTo(map);
    clusterGroups[i]=group;
  }
}
PLAIN ENGLISH

A counter that ticks up every time a render starts. It lives outside the function so it survives between renders.

The entry point. This is what runs when the user clicks "Markers."

Capture my number before I start work. If a newer render comes in, it'll bump the counter past mine — and I'll know to bow out.

Do 5,000 markers per slice. Picked empirically: big enough that the loop overhead is negligible, small enough that one slice finishes in a few milliseconds.

Loop over the three categories (UFO, Bigfoot, Haunted). Skip any that have no filtered data.

Set up a Leaflet cluster group to hold this category's pins.

The inner worker function. This is the one that gets called over and over.

Am I stale? If someone newer has started, abandon the rest of my work immediately. No cleanup needed.

Grab the next 5,000 markers (or whatever's left).

Build them into a temporary array, then add them all at once — one DOM touch instead of 5,000.

Still more to do? Queue up the next batch with setTimeout. The 0 means "right after the browser catches up on everything else."

Kick it off the first time, attach the cluster to the map, and stash a reference so we can remove it later.

Screen 3 — The other five tricks

Batching is just one illusion in the magician's bag. The rest of the app leans on five more, each suited to a different kind of heavy lifting. Learn their names — the next time an AI-built app feels janky, you'll know which one to ask for.

1

Batched rendering

What: Chop a big loop into 5,000-item slices with a setTimeout between each.

When to ask for it: any loop over 10K+ items that runs on the main thread.

2

Spatial index caching

What: Build an expensive spatial index once, reuse it on every hover and recount.

When to ask for it: the same input keeps being asked the same question (hover-hover-hover).

3

Canvas renderer

What: Tell Leaflet preferCanvas: true. Pins become pixels on one canvas, not 258K DOM elements.

When to ask for it: drawing thousands of small graphical things.

4

MarkerCluster

What: A Leaflet plugin that lumps nearby pins into one circle-with-a-count and only renders what's visible at this zoom.

When to ask for it: data density is way higher than the screen's pixel density.

5

Debounced zoom

What: Ignore every zoomend event except the last — wait for 300ms of quiet before re-rendering.

When to ask for it: an event fires in rapid bursts and each run is expensive.

6

Web Worker

What: Hand the hex-count math to hex-worker.js — a parallel thread — so the main thread stays free to paint.

When to ask for it: pure math work that takes more than ~50ms.

Trick #5 in the wild — the debounced zoom

Scroll-zooming fires zoomend ten times in a second. Re-rendering a heatmap for every single one would be murderous. The fix is five extra lines:

CODE 
let _zoomRenderTimer=null;
map.on('zoomend',()=>{
  document.getElementById('stat-zoom').textContent=map.getZoom();
  if(currentView==='heatmap'||currentView==='hexbin'){
    if(_zoomRenderTimer)clearTimeout(_zoomRenderTimer);
    _zoomRenderTimer=setTimeout(()=>{_zoomRenderTimer=null;renderCurrentView();},300);
  }
});
PLAIN ENGLISH

A slot to remember the pending re-render we've scheduled but not yet run.

Leaflet fires zoomend every time a zoom completes. Scroll-zooming fires it repeatedly — once per wheel tick.

Update the zoom number in the sidebar (this is cheap, always do it).

Only the expensive views — heatmap and hexbin — need the debounce dance.

If we already had a re-render queued, cancel it. We're about to queue a fresher one.

Wait 300ms. If no new zoom arrives, fire renderCurrentView. If another zoom does come in, this timer gets cancelled on the next line and replaced with a new 300ms countdown.

Trick #7 — The stale render problem

Here's the subtle bit — the one AI almost always forgets. Batched rendering takes time. While one render is mid-way through its 52 slices, what happens if the user clicks "apply filter" again, with different inputs? Now there are two batch sequences running, both writing to the same cluster groups. Pins from the old filter mingle with pins from the new one. Chaos.

The fix is one integer: markerRenderGen. Every new render bumps the counter and captures the old value as myGen. At the top of every batch, it checks: "Am I still the newest render?" If a newer one started, the stale batch quietly bows out. No cleanup. No coordination. Just one number.

1
First click lands. markerRenderGen goes 0 → 1.

Render A grabs myGen = 1, starts chewing through 52 batches.

2
Halfway through, the user clicks again. markerRenderGen goes 1 → 2.

Render B grabs myGen = 2, starts its own 52 batches.

3
Render A's next batch wakes up.

It checks: markerRenderGen (2) !== myGen (1). It returns. Silently. Its remaining batches never run.

4
Render B finishes clean.

Only the newest filter's pins make it to the map. No mingling. No bug report.

!
Aha! Long async work needs a stale-check.

Any time you kick off work that spans multiple ticks of the event loop, you need a way to detect it's been superseded. A generation counter is the cheapest version of that idea — one integer, one check per slice, done. Whenever you ask AI for batched, debounced, or streamed work, also ask: "What happens if a new one starts before the old one finishes?"

Bonus — what caching looks like

Trick #2 (the spatial index) is a one-function example of caching done right. Build it once. Hand it back on every future call. The savings show up across thousands of hex recounts.

CODE 
function buildSpatialIndex(cellDeg){
  if(cachedSpatialIndex)return cachedSpatialIndex;
  const idx={};
  for(let cat=0;cat<3;cat++){
    filteredCat[cat].forEach(r=>{
      const key=Math.floor(r[F.LAT]/cellDeg)+','+Math.floor(r[F.LON]/cellDeg);
      (idx[key]=idx[key]||[]).push(r);
    });
  }
  cachedSpatialIndex=idx;
  return idx;
}
PLAIN ENGLISH

Take a grid-cell size in degrees as input.

Already built? Return the cached version. This is the whole point — don't rebuild.

Fresh build: start with an empty lookup table.

For each of the three categories...

...walk every point in that category...

...compute a grid-cell key like "395,-845". Points in the same cell get the same key — a cheap way to group by location.

Append this point to the bucket for that key. Create the bucket if it's the first point there.

Stash the finished index. Next call gets it for free.
