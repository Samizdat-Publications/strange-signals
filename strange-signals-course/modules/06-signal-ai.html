SIGNAL AI — The Agent Loop

How a text-only model gets hands — 80 lines of JavaScript that turn Claude into an air-traffic controller for your map.

What just happened?

You type "show me UFO hotspots near military bases in the 90s" into SIGNAL's chat box. Three seconds later the map has zoomed, the year filter is 1990–1999, the military overlay is on, and the heatmap is rendering. You didn't click anything.

Picture an LLM as an air-traffic controller in a tower with binoculars and a radio. The controller cannot fly the planes — all they can do is see (look through the binoculars) and talk (issue instructions over the radio). Strange Signals' AI is the same: Claude cannot touch the map directly. It can only ask to see something or do something, and the browser runs each request and reports back.

Those requests are called tools. Strange Signals exposes about two dozen of them, grouped into four flavors:

eye

See the map

Read what's currently visible — bounds, zoom, which categories are on, how many records survive the filters. Example: get_map_state.

drive

Drive the map

Change what's visible — pan, zoom, switch view, apply filters, toggle overlays. Example: set_view, apply_filters.

math

Analyze

Run the Module 5 math on demand — correlations, clusters, nearest-neighbor distances, report builders. Example: compute_correlation.

pin

Annotate

Stick pins on the map with notes and colors; draw highlight zones the user can see. Example: add_annotation.

Every one of those is just a plain JavaScript function with a JSON description attached. Claude reads the descriptions, decides which to call, and supplies the arguments.

Back to step 2. Claude sees its own request plus the result, and decides whether to call more tools or finish with text. The tool-use loop caps at 10 turns.

!
Aha! Tool results go back as a "user" message.

A common bug in hand-rolled loops: developers assume tool results are their own message role. They are not — the API wraps them as the user turn that answers the assistant's tool_use. Miss this and Claude gets confused about who said what.

The fetch call, demystified

There is no backend. SIGNAL's entire "server" is this one function — the browser talks to Anthropic directly, using the user's own API key stored in localStorage. Here is the setup half of the loop:

CODE 
async function runConversationLoop(){
  var apiKey=getApiKey();
  var model=getModel();

  // Mark system prompt + last tool as cacheable (ephemeral, ~5 min TTL)
  var cachedSystem=[{type:'text',text:SYSTEM_PROMPT,cache_control:{type:'ephemeral'}}];
  var cachedTools=TOOLS.map(function(t,i){
    return i===TOOLS.length-1 ? Object.assign({},t,{cache_control:{type:'ephemeral'}}) : t;
  });

  for(var turn=0;turn<10;turn++){
    var body={
      model:model, max_tokens:4096,
      system:cachedSystem, tools:cachedTools,
      messages:messages, stream:true
    };

    var resp=await fetch('https://api.anthropic.com/v1/messages',{
      method:'POST',
      headers:{
        'Content-Type':'application/json',
        'x-api-key':apiKey,
        'anthropic-version':'2023-06-01',
        'anthropic-dangerous-direct-browser-access':'true'
      },
      body:JSON.stringify(body)
    });
    // (retry on 429/529 with retry-after header omitted for clarity)

    var result=await parseStream(resp);
    // ... continued in the next snippet
  }
}
PLAIN ENGLISH

Grab the API key the user pasted in (stored in localStorage) and the model they picked from a dropdown.

Mark the system prompt as cacheable. This is prompt caching — Anthropic will hold this text in cache for about five minutes so you are not charged for it every turn.

Do the same for the last entry in the tool catalog. Marking one tool automatically caches everything up to and including it.

Loop up to ten times. Hard cap — enough for multi-step plans but bounded so a confused model cannot run away with your bill.

Build the request body. The whole conversation goes in the messages array every single time — Claude has no memory between calls, so the array is the memory.

Streaming on, so tokens show up in the chat as they arrive.

Call fetch straight to Anthropic. No backend. The browser is the server.

The API key goes in the x-api-key header. Whoever opens DevTools on this page can see it — fine for a personal tool, never for a shipped product.

The anthropic-dangerous-direct-browser-access header is the opt-in flag that says "yes, we really are calling the API straight from the browser." Without it, CORS would block the request.

Serialize the body to JSON and send it.

In real life there is also a retry block handling 429 / 529 errors — if the API says "too fast," the app sleeps and tries again.

Parse the streamed response into a clean object with a list of content blocks. Text blocks or tool_use blocks. That is what goes to the next snippet.

The loop body: executing tools

Now the second half — what the browser does after Claude replies. This is where "agent" stops being mystical: it is an if, a for, and two array pushes.

CODE 
var result=await parseStream(resp);
var hasToolUse=result.content.some(function(b){return b.type==='tool_use'});

messages.push({role:'assistant',content:result.content});
saveConversation();

if(!hasToolUse)break;

// Execute tool calls
var toolResults=[];
for(var i=0;i<result.content.length;i++){
  var block=result.content[i];
  if(block.type!=='tool_use')continue;
  try{
    var toolResult=await executeTool(block.name,block.input);
    toolResults.push({type:'tool_result',tool_use_id:block.id,content:JSON.stringify(toolResult)});
  }catch(e){
    toolResults.push({type:'tool_result',tool_use_id:block.id,content:JSON.stringify({error:e.message}),is_error:true});
  }
}

messages.push({role:'user',content:toolResults});
saveConversation();
PLAIN ENGLISH

Parse the streamed response. Claude's reply is not a single string — it is a list of blocks. Text, tool_use, sometimes thinking blocks.

Ask: did Claude request any tools? If even one block has type === 'tool_use', we have work to do.

Save Claude's entire turn onto the conversation history, tool requests and all.

Write the updated messages array to localStorage. If the user refreshes the page, the conversation comes back.

If Claude's reply was pure text, there is nothing to execute — break out of the loop and the user sees the final answer.

Otherwise, prepare a list to collect tool results in.

Walk every block. Skip anything that is not a tool request (text blocks are passed through).

For each tool request, call executeTool(name, input). This is the one function that knows how to actually drive the map — a giant switch statement over tool names.

Wrap the return value in a tool_result block, keyed by the original request's id. Claude uses the id to pair "I asked for X" with "here is the answer to X" — critical when multiple tools fire in parallel.

If a tool throws, send the error back tagged is_error: true. Claude reads the error and can try a different approach. Robust-agent behavior — one bad call doesn't crash the chat.

Here is the pairing from the Aha callout — tool results go back as a user turn, not assistant. That is how the API expects them.

Persist again. Then the outer for loops back to the fetch, Claude sees the results, and decides what to do next.

!
Aha! Prompt caching is a 2-line optimization.

Look back at cache_control: {type: 'ephemeral'} in the previous snippet. That single flag tells Anthropic "the system prompt and tool list don't change across turns — hold them for about five minutes." After the first call, subsequent calls get charged a fraction of the input tokens for the cached prefix. Tiny code change, large bill savings on multi-turn chats.

The whole point of this course is to make you a sharper steerer when AI writes code for you. Agent loops are where most "AI assistant" features go wrong. Three scenarios, one right answer each.

AI built you a "chat with your app" feature. When you test it, the assistant's first reply says "I called the weather API" — but no API call actually happened, and the app doesn't expose a weather tool. What's the most likely bug?

Your AI agent runs one step, finishes, and the user's follow-up question has no memory of what was just said. What's the most likely cause?
