The Pipeline

From five scattered CSVs to 258,000 pins in your browser — without freezing the page.

The journey, in one picture

The dataset for this app is 258,000 records. A naive JSON file of all of them is around 70 MB. Dropped raw onto a page, you'd stare at a frozen tab for half a minute. Strange Signals loads it in a handful of seconds, and the page never freezes — even during parsing.

Think of it like a shipping warehouse the week before a holiday sale. Goods arrive from five different suppliers in five different formats. They get sorted, labeled, packed efficiently, loaded onto one truck, and delivered. Then a helper on the receiving end unloads the truck in the background — the shopkeeper never has to stop serving customers.

The punchline

The first five stops all happen on a developer's laptop before the app is ever shipped. Only the last two — downloading the JSON and parsing it — happen on your machine.

Trick #1: the packing trick

Imagine labeling every single box in the warehouse with the words "this is a box" on all six sides. Useful? No. Wasteful? Absolutely. That's what a verbose JSON record does — it repeats the column names on every single row.

With 258,000 records, repeating "latitude" and "longitude" and "category" over and over adds up to many megabytes of dead weight. So this app strips the labels and stores every record as a plain list, knowing the code on the other end can figure out the order.

A

Verbose (what most AI writes first)

{ "latitude": 39.12,
  "longitude": -84.56,
  "category": "UFO/UAP",
  "date": "2020-01-15",
  "location": "Cincinnati, OH" }

Every column name written out every time. Clear to read, huge on disk.

B

Compact (what Strange Signals does)

[39.12, -84.56, 0,
 "2020-01-15",
 "Cincinnati, OH",
 "triangle",
 "Bright light..."]

A plain list. Position 0 is always latitude. Position 1 is always longitude. Roughly half the file size.

!
Aha! Array positions are the column names.

The code knows position 0 = lat, position 1 = lon, position 2 = category. The F = {LAT:0, LON:1, ...} cheat sheet from Module 2 is exactly what reads these lists. The column names live in one place — in the code — not 258,000 times on disk.

How the packing actually happens

Here's the Python that does the squeezing. It runs once, on the developer's laptop, when the data pipeline rebuilds.

CODE 
# Build records as list of lists (vectorized via itertuples — 10x faster than iterrows)
records = []
for r in df.itertuples(index=False):
    records.append([
        float(r.latitude),
        float(r.longitude),
        int(r.cat),
        r.date_str if r.date_str != "nan" else "",
        r.loc,
        r.sub if r.sub != "nan" else "",
        r.desc if r.desc != "nan" else "",
    ])

output = {
    "categories": ["UFO/UAP", "Bigfoot/Sasquatch", "Haunted Place"],
    "fields": ["lat", "lon", "cat", "date", "location", "subcategory", "description"],
    "data": records,
}
PLAIN ENGLISH

Start with an empty list. Every sighting in the warehouse gets added to it.

Walk every row of the cleaned DataFrame one at a time. Using vectorized iteration makes this pass fast.

For each row, throw away the column names. Keep the values in a fixed order: latitude, longitude, category code, date, location, subcategory, description.

Empty strings replace missing values so downstream code never has to check for "nan".

Wrap the giant list in a tiny header that tells the browser what each position means. This header is the decoder ring — fields[0] is "lat", so data[i][0] is the latitude of record i.

Trick #2: the unfreeze trick

Back to the warehouse. Your main thread is the shopkeeper at the counter. Smiling at customers. Ringing things up. If the shopkeeper personally unloads the delivery truck, every customer waits while they sweat through 258,000 boxes.

So you hire a dock worker — a Web Worker. They open the truck around back, sort the pallets into three bins (UFO, Bigfoot, Haunted), and pass finished boxes up to the counter in small batches. The shopkeeper keeps smiling. Nobody notices the unloading.

1
Validate and categorize in one pass

The worker throws out any record with a bad latitude, longitude, or category code — and groups the survivors into three lists (one per category).

2
Ship results in 5,000-record batches

Each small chunk crosses the thread boundary via postMessage. Small chunks copy fast. A giant blob would cause a visible hiccup on the main thread.

The entire worker, in 30 lines

CODE 
// Web Worker: decode, parse, validate, categorize — all off main thread.
// Sends results back in small batches so the main thread never blocks on structured clone.
self.onmessage=function(e){
  try{
    const text=new TextDecoder().decode(e.data);
    const json=JSON.parse(text);
    const raw=json.data;
    if(!Array.isArray(raw)||!raw.length){
      self.postMessage({type:'error',error:'Invalid or empty data'});
      return;
    }
    // Filter + categorize in one pass
    const cats=[[],[],[]];
    for(let i=0;i<raw.length;i++){
      const r=raw[i];
      if(!Array.isArray(r)||r.length<7)continue;
      if(typeof r[0]!=='number'||isNaN(r[0]))continue;
      if(typeof r[1]!=='number'||isNaN(r[1]))continue;
      if(r[2]<0||r[2]>2)continue;
      cats[r[2]].push(r);
    }
    const total=cats[0].length+cats[1].length+cats[2].length;
    // Send back in small batches to avoid structured clone blocking main thread
    const BATCH=5000;
    for(let cat=0;cat<3;cat++){
      for(let i=0;i<cats[cat].length;i+=BATCH){
        self.postMessage({type:'batch',cat:cat,records:cats[cat].slice(i,i+BATCH)});
      }
    }
    self.postMessage({type:'done',total:total});
  }catch(err){
    self.postMessage({type:'error',error:err.message});
  }
};
PLAIN ENGLISH

This worker sits quietly and waits for the main thread to hand it a file. When that happens, this function runs.

Turn the raw bytes into text, then turn the text into a JavaScript object.

Sanity check: is there actually data here? If not, send an error back and bail out.

Three empty buckets — one per category (UFO, Bigfoot, Haunted).

Walk every record. Skip the broken ones: wrong shape, bad latitude, bad longitude, or a category code outside 0 through 2.

Drop each valid record into the bucket named by its category code. All three buckets fill up in a single pass.

Now ship results back — 5,000 records at a time.

Each chunk is small enough that the browser's structured clone copy is invisible to the user.

When the last batch is out the door, send a "done" message with the final tally.

If anything blew up, send the error back to the main thread instead of crashing silently.

What the main thread actually does

Now that the worker is doing the heavy lifting, the shopkeeper's job is almost embarrassingly small: receive a box, drop it in the right bin, update the progress bar, smile at the next customer. Repeat.

1
A batch message arrives

The worker calls postMessage({type:'batch', cat:0, records:[...5000 records...]}). The main thread's listener wakes up for a fraction of a second.

2
Append to the right bucket

The main thread has three arrays in memory: catArrays[0] (UFO), catArrays[1] (Bigfoot), catArrays[2] (Haunted). It pushes the batch onto the one matching cat. That's it.

3
Nudge the progress bar, then back to idle

The UI updates. The main thread returns to its normal duties — handling clicks, scrolls, and redraws — until the next batch lands.

4
"done" arrives, the map draws

When the worker signals completion, the main thread kicks off the rendering pipeline — which is Module 4's whole job.

Two tricks every big-data browser app should use

{ }

Compact array format

Store records as plain lists, not objects with repeated keys. Half the file size. The field-order header is your decoder ring.

W

Web Worker + batched postMessage

Parse and validate off the main thread. Ship results back in small chunks so the copy across threads is never a visible hiccup.

!
Why this matters when you're using AI

Ask AI to "load a big JSON file in the browser" and by default it'll hand you fetch(url).then(r => r.json()) on the main thread — which freezes the UI for big files. Now you know the names to ask for: compact array format and parse it in a Web Worker with batched postMessage. That vocabulary is the difference between a snappy app and a frozen tab.
