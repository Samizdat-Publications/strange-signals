# Signal Intelligence Feature Set — Design Spec

**Date:** 2026-03-17
**Status:** Draft
**Author:** Claude (brainstorming with Stewart)

## Goal

Transform Strange Signals from a manual analysis tool into an AI-driven investigation platform. Users interact primarily through a conversational AI assistant ("SIGNAL") that can control every aspect of the map, run analyses, highlight findings, and explain results — all through natural language.

## Scope

Three sub-projects, built in order:

1. **Draggable/Resizable Window System** — shared infrastructure for all floating panels
2. **Correlation Performance + UI Polish** — Web Worker for hex binning, progress indicators
3. **AI Assistant ("SIGNAL")** — Claude-powered conversational interface with tool use

Out of scope for this spec: additional data source ingestion, PDF report generation (future extensions once the assistant exists).

## Sub-Project 1: Window Manager

### Problem
The temporal correlation overlay is a fixed-position `<div>` that can't be moved or resized. The matrix results panel is inline in the sidebar. Users requested draggable, resizable panels for all modal-like UI elements.

### Design

A lightweight `WindowManager` class (new file: `window-manager.js`) that provides:

- **Drag** via title bar mousedown/touchstart
- **Resize** via a corner handle (bottom-right)
- **Minimize** to a small icon strip at the bottom of the map
- **Close** (hide)
- **Z-index stacking** — clicking a window brings it to front
- **Snap to edges** — optional, prevents windows from going off-screen
- **Remember position** — stores last position in localStorage per window ID

#### API

```js
// Create a managed window
const win = WindowManager.create({
  id: 'temporal',
  title: 'TEMPORAL CORRELATION',
  content: document.getElementById('temporal-content'),
  defaultPos: { right: 20, bottom: 60 },
  defaultSize: { width: 720, height: 400 },
  minSize: { width: 400, height: 250 },
  onClose: () => { /* cleanup */ }
});

// Programmatic control
win.show();
win.hide();
win.minimize();
win.focus();
win.setTitle('NEW TITLE');
```

#### HTML Structure (generated by WindowManager)

```html
<div class="ss-window" id="win-temporal" style="...">
  <div class="ss-window-titlebar">
    <span class="ss-window-title">TEMPORAL CORRELATION</span>
    <div class="ss-window-controls">
      <button class="ss-win-btn minimize" title="Minimize">&#9472;</button>
      <button class="ss-win-btn close" title="Close">&times;</button>
    </div>
  </div>
  <div class="ss-window-body">
    <!-- content injected here -->
  </div>
  <div class="ss-window-resize-handle"></div>
</div>
```

#### Styling
- Matches existing dark sci-fi aesthetic (same `--bg-panel`, `--border` vars)
- Title bar: Orbitron font, subtle glow on focus
- Resize handle: small triangle in bottom-right corner
- Minimized strip: row of small labeled buttons at bottom of map container
- Transitions: smooth opacity/transform for show/hide/minimize

#### Migration
- Temporal overlay: extract inner content from `#temporal-overlay`, create window
- Matrix results: move out of sidebar into a window (triggered by matrix-run button)
- Chat panel: uses the same system (Sub-Project 3)

### Files
- **Create:** `window-manager.js` (~200 lines) — WindowManager class
- **Create:** `window-manager.css` (~80 lines) — window chrome styles
- **Modify:** `index.html` — add script/css refs, restructure temporal overlay
- **Modify:** `strange-signals.js` — replace temporal show/hide with WindowManager calls
- **Modify:** `strange-signals.css` — remove old temporal-overlay positioning styles

## Sub-Project 2: Correlation Performance

### Problem
Switching to CORR view on 183K records takes 1-2 minutes and the UI appears frozen. The bottleneck is `getOrBuildHexData()` which runs Turf.js `booleanPointInPolygon()` for each record against each hex cell on the main thread.

### Design

#### Web Worker for Hex Binning

Move the heavy computation into a Web Worker (`hex-worker.js`):

```
Main Thread                          Worker Thread
-----------                          -------------
postMessage({                        onmessage: (data) => {
  points: [...],                       // Build hex grid (turf.hexGrid)
  cellSide, bounds                     // Spatial index + point assignment
})                                     // Permutation p-values
                                       postMessage({ grid, counts, stats })
onmessage: (result) => {             }
  // Render hex layer
  // Update UI
}
```

**Key constraints:**
- Turf.js must be loaded in the worker via `importScripts()`
- D3 is NOT needed in the worker (only used for rendering)
- Worker sends progress messages so the main thread can show a progress bar
- Cache invalidation still works — we cache the worker result, invalidate on filter change

#### Progress Indicator

When correlation is computing:
1. Disable the "RUN ANALYSIS" button, show spinner
2. Show a progress bar in the correlation panel (reuse existing `.progress-track` style)
3. Display estimated stage: "Building hex grid..." → "Assigning 183K points..." → "Computing correlations..." → "Running significance tests..."

#### Message Protocol

```js
// Main → Worker
{ type: 'compute', points: Float64Array, cellSide: 25, bounds: [...], categories: 3 }

// Worker → Main (progress)
{ type: 'progress', pct: 45, stage: 'Assigning points to hexes...' }

// Worker → Main (result)
{ type: 'result', grid: GeoJSON, counts: [...], elapsed: 1234 }
```

Using `Float64Array` transferable for the points data avoids copying 183K records.

### Files
- **Create:** `hex-worker.js` (~150 lines) — Web Worker for hex computation
- **Modify:** `strange-signals.js` — replace inline hex computation with worker calls, add progress UI
- **Modify:** `index.html` — add progress bar elements to correlation panels

## Sub-Project 3: AI Assistant ("SIGNAL")

### Architecture Overview

```
User types message
       |
       v
  [Chat UI]  ──→  [Anthropic Messages API]
       |                    |
       |              Claude responds with
       |              tool_use blocks
       |                    |
       v                    v
  [Display text]    [Execute tools on map]
       |                    |
       |              Tools return results
       |                    |
       v                    v
  [Continue conversation with tool results]
```

### API Integration

**Direct browser → API** (no backend proxy). User enters API key in a settings modal, stored in `localStorage`.

- **Model:** `claude-sonnet-4-6` (fast, cost-effective for tool use)
- **Endpoint:** `https://api.anthropic.com/v1/messages`
- **Headers:** `x-api-key`, `anthropic-version: 2023-06-01`, `anthropic-dangerous-direct-browser-access: true`
- **Streaming:** Yes, via SSE for real-time response rendering

#### CORS Note
The Anthropic API supports direct browser access with the `anthropic-dangerous-direct-browser-access: true` header. This is the officially supported path for client-side applications.

### Highlight System

The visual highlighting uses Leaflet's custom layers with CSS animations:

#### Phase 1: Radar Ping (0-3 seconds)
Three concentric rings expand outward from the center point with decreasing opacity. Uses CSS `@keyframes` animation on SVG circle overlays.

```css
@keyframes radar-ping {
  0% { r: 0; opacity: 0.8; }
  100% { r: 80; opacity: 0; }
}
```

Each ring is staggered by 0.5s delay. Color matches the `color` parameter from `highlight_areas`.

#### Phase 2: Persistent Spotlight (after 3 seconds)
The ping settles into a semi-transparent filled circle with:
- Soft radial gradient (bright center → transparent edge)
- Numbered label in Orbitron font (matches theme)
- Subtle pulsing glow animation (slow, 3s cycle)
- Click to dismiss individual spotlights
- All spotlights clear when a new query is sent or user clicks "Clear highlights"

#### Implementation
A custom Leaflet layer class `L.HighlightLayer` that manages the SVG overlay. Each highlight is an SVG group with the animated circles + label text.

### Conversation Flow (Multi-turn with Tool Use)

```
1. User: "Show me Bigfoot hotspots in Ohio that correlate with UFOs"

2. JS → API: messages=[{role:'user', content:'...'}], tools=[...]

3. Claude responds: tool_use blocks:
   - set_filters({ state: 'OH', categories: [0, 1] })
   - run_spatial_correlation({ category_a: 0, category_b: 1, hex_size_km: 25 })

4. JS executes tools, sends results back:
   - set_filters result: { visible: 2341, filtered: true }
   - correlation result: { r: 0.42, pValue: 0.003, nHexes: 47 }

5. Claude responds with more tool_use:
   - detect_clusters({ min_sightings: 10, hex_size_km: 25 })

6. JS executes, sends result:
   - clusters: [{ centroid: [40.1, -82.5], total: 145, ... }, ...]

7. Claude responds with final tool_use + text:
   - highlight_areas({ areas: [{ lat: 40.1, lon: -82.5, label: '1', ... }, ...] })
   - zoom_to_region({ state: 'OH', zoom: 7 })
   - Text: "I found 3 significant hotspots in Ohio..."

8. JS highlights areas, zooms map, displays text response
```

### Token Management

- System prompt: ~300 tokens
- Tool definitions: ~1200 tokens
- Per message round-trip: ~500-2000 tokens depending on tool results
- **Conversation cap:** Keep last 20 messages (user + assistant). Older messages are trimmed from the front. No summary generation — just drop oldest messages. The system prompt provides enough context for Claude to work with partial history.
- **Tool result truncation:** `get_sightings_in_area` limits to 20 records. Cluster detection returns top 10 clusters. This keeps tool results under ~1000 tokens each.

  // Filters
  applyFilters: applyFilters,

## Build Order

1. **Window Manager** (no dependencies)
2. **Hex Worker + Progress UI** (no dependencies, but benefits from window manager for result panels)
3. **Highlight Layer** (depends on Leaflet only)
4. **AI Assistant** (depends on all above: windows for chat panel, worker for fast analysis, highlights for map callouts)

Each sub-project is independently testable and committable.
